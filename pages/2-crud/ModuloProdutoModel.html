ModuloProdutoModel novo arquivo

O Model é a camada de regras de negócio — fica entre o formulário e o DAO. Neste CRUD simples não há regras, por isso o arquivo é pequeno: insert e update apenas delegam para o DAO.

package br.xt.app.modulo.produto;

import br.jasap.core.AppManager;

public class ModuloProdutoModel extends ModuloProdutoDAO {

    public ModuloProdutoModel(AppManager manager) {
        super(manager);
    }

    public void insert(ModuloProdutoBean bean) throws Exception {
        daoInsert(bean);
    }

    public void update(ModuloProdutoBean bean) throws Exception {
        daoUpdate(bean);
    }

}

Por que o Model é tão pequeno?

O papel do Model é concentrar as regras de negócio que precisam rodar antes de persistir um dado. Em módulos mais complexos, o insert() e o update() fariam coisas como:

  • Calcular campos derivados
  • Disparar cascata em outras tabelas
  • Validar consistência de dados
  • Chamar outros Models

Neste CRUD não há nada disso — então o Model só delega para o DAO.

Você pode estar se perguntando: "por que não chamar o DAO direto do formulário?" — A resposta é separação de responsabilidades. O formulário chama o Model, não o DAO. Se amanhã precisar adicionar uma regra no insert, ela entra no Model sem tocar no formulário nem no DAO.

extends ModuloProdutoDAOHerda todos os métodos do DAO (daoList, daoSingle, daoInsert, daoUpdate, daoDelete). O Model acessa o DAO diretamente pela herança, sem precisar criar uma instância

ModuloProdutoModel(AppManager manager)

Passa o AppManager para o construtor do DAO via super(manager). É quem inicializa a conexão com o banco — sem isso os métodos herdados do DAO não teriam acesso ao banco de dados.

insert(bean) / update(bean)

Por que apenas insert e update, e não delete?

Delete não tem regra de negócio neste módulo — o formulário chama daoDelete() diretamente. Insert e update existem no Model porque são as operações onde regras de negócio tendem a aparecer conforme o módulo evolui.
